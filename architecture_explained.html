1. What is IntelCrypt?

High-level project overview

IntelCrypt is a full-stack, enterprise-grade secure messaging platform designed for classified communications. It is built as a client-server system where the Flutter mobile app acts as the client and a Spring Boot REST API acts as the server. All messages are encrypted end-to-end using military-grade cryptography before leaving the device, meaning even the server cannot read message content.

🏗️ Architecture Style
  • Clean Architecture — strict layer separation
  • Client–Server model (Flutter ↔ Spring Boot)
  • REST API + WebSocket (STOMP) for real-time
  • JPA/Hibernate ORM for database abstraction
  • Riverpod for reactive state management (frontend)
🔐 Core Security Principle
  • Zero plaintext stored — all content encrypted
  • End-to-end encryption: AES-256-GCM + RSA-2048
  • JWT tokens for stateless authentication
  • Argon2id password hashing (never bcrypt-plain)
  • Full audit trail — every action logged with IP + UA

1
📲 App Installation
The user downloads IntelCrypt from the Google Play Store (Android) or Apple App Store (iOS). The Flutter app supports both platforms from a single codebase.
2
📝 User Registration — /api/auth/register
The user submits their username, email, and password. On registration, the server:
  • Hashes the password using Argon2id (memory-hard, brute-force resistant)
  • Generates an RSA-2048 key pair — public key stored on the server, private key encrypted with the user's password-derived key (PBKDF2) and stored encrypted
  • Returns a JWT access token + refresh token for the session
  • Writes an AUDIT_LOG entry recording the registration IP and user-agent
3
🔒 Permission Handling
The Flutter app requests three OS-level permissions needed for advanced features:
• Biometric — fingerprint / Face ID via local_auth package, backed by Android Keystore / iOS Keychain
• Location — used for location-based access verification (LocationVerificationService)
• Microphone — used for acoustic environment fingerprinting and voice note recording
4
👥 Role Selection & Assignment
Every user is assigned a Role and a Clearance Level which controls what they can do:
• ROLE_USER — standard messaging access
• ROLE_CLASSIFIED — can send confidential/secret-level messages
• ROLE_ADMIN — full admin dashboard, user management, audit log access
Clearance levels: STANDARD → CONFIDENTIAL → SECRET → TOP_SECRET

3. Secure Messaging Flow

How a message travels from sender to recipient — encryption at every step

💡
Key principle: Zero-knowledge server The server never sees plaintext. Message content is encrypted on the sender's device before transmission and can only be decrypted by the intended recipient using their private key.
1
🖊️ Compose Message
The user types a message in the Chat Message Screen. The Flutter MessageInputField widget captures the plaintext locally — it never leaves the device in plaintext form.
2
🔒 AES-256-GCM Encryption (client-side)
The EncryptionService on the Flutter side (using PointyCastle) encrypts the message:
  • A random 256-bit AES key is generated per message
  • A random 96-bit IV (Initialization Vector) is generated
  • The message is encrypted using AES-256-GCM mode which produces: IV || Ciphertext || Authentication Tag
  • The Auth Tag ensures the message hasn't been tampered with (integrity protection)
3
🔏 RSA-2048 Key Wrap (Hybrid Encryption)
The AES content key cannot be sent as-is. Instead:
  • The sender fetches the recipient's public RSA-2048 key from /api/keys
  • The AES content key is encrypted with the recipient's public key using RSA-OAEP padding
  • This "wrapped" key is attached to the message as encryptedContentKey
  • Only the recipient's private key (which only they have) can unwrap it
4
📤 Send via HTTPS — POST /api/messages/send
The encrypted payload is sent over HTTPS with a JWT bearer token. The request contains:
{ encryptedContent, encryptedContentKey, encryptionAlgorithm, classification }
The server stores this in the messages table — no plaintext is ever written to the database.
5
🔔 Push Notification
The NotificationService on the recipient's device wakes the app. The notification body only shows metadata (sender name, time) — the actual message content is never in the notification payload.
6
🔓 Recipient Decrypts
The recipient opens the chat. The app fetches the encrypted message and:
  • Loads the recipient's encrypted private key from secure storage (Android Keystore / iOS Keychain)
  • Unlocks it using the user's password or biometric
  • Uses the private RSA key to decrypt the AES content key
  • Uses the AES key + IV to decrypt the ciphertext and verify the Auth Tag
  • The decrypted plaintext is displayed in the MessageBubble widget
7
✅ Delivered + Read Receipt
The message status progresses through four states: SENT → DELIVERED → READ. The read timestamp is recorded via PUT /api/chats/{chatId}/messages/{id}/read.

5. Database & API Layer

How the Spring Boot server stores and retrieves data

🌐 REST API Endpoints
  • /api/auth/* — register, login, refresh, logout, change-password
  • /api/chats/* — CRUD, direct DM, group, participants, search
  • /api/messages/* — send, inbox, sent, decrypt, delete, ack
  • /api/chats/{id}/messages — per-chat paginated message fetch
  • /api/keys/* — generate, rotate, status check
  • /api/users/* — profile, search, update
  • /api/files/* — encrypted attachment upload/download
  • /api/admin/* — audit logs, user management (ADMIN only)
  • /api/health — server health check (public, no auth)
🔌 WebSocket / STOMP
  • Real-time messaging without polling
  • STOMP protocol over WebSocket
  • StompChannelInterceptor validates JWT on every WS frame
  • WebSocketService on Flutter subscribes to chat topics
  • Used for: message delivery, read receipts, typing indicators, presence (online/offline status)
🗄️
Database: H2 (dev) → PostgreSQL (prod) In development the app uses H2 in-memory database (zero setup). In production it switches to PostgreSQL. All 6 JPA entities (User, Message, Chat, EncryptionKey, AuditLog, Attachment) are mapped via Hibernate with optimized indexes on sender_id, recipient_id, and message expiry columns for fast queries.

6. Security Filter Chain

Every HTTP request passes through this chain before reaching any controller

Incoming Request
→
🚦 Rate Limiter
→
🐝 Honeypot
→
🪙 JWT Filter
→
⚙️ Controller
→
📋 Audit Log
1
🚦 RateLimitFilter
Uses a token bucket algorithm per IP address. Limits requests to 100 per 60 seconds by default (configurable in application.properties). If exceeded, returns HTTP 429. Prevents brute-force attacks on login, key endpoints, and message APIs.
2
🐝 HoneypotFilter
Contains fake decoy API endpoints that look real to attackers (e.g., fake admin endpoints, fake data dumps). Any request to these endpoints immediately flags the IP as suspicious, logs the event as CRITICAL severity in AuditLog, and can trigger an IP block. Legitimate app users never hit these endpoints.
3
🪙 JwtAuthenticationFilter
Extends Spring's OncePerRequestFilter — runs exactly once per request:
  • Extracts Bearer {token} from the Authorization header
  • Verifies the JWT signature, expiry, and claims using JwtTokenService
  • Loads the UserDetails from the database and sets the SecurityContextHolder
  • Public endpoints (/api/auth/login, /api/auth/register, /api/health, Swagger UI) are skipped
4
⚙️ Spring Controllers + @PreAuthorize
Once authenticated, method-level authorization is enforced. Classified message endpoints require @PreAuthorize("hasAnyRole('CLASSIFIED', 'ADMIN')"). Admin endpoints require ROLE_ADMIN. Regular chat/message endpoints require just a valid JWT (any registered user).
5
📋 AuditService — Full Audit Trail
Every significant action writes an AuditLog record: login, logout, message sent, key rotated, password changed, failed login, admin action — all with timestamp, IP address, user-agent, event type, and severity (INFO / WARN / CRITICAL).

7. Advanced Auth & Unique Features

Features that go beyond standard secure messaging

🎙️ Acoustic Fingerprint Authentication
  • Records the ambient acoustic environment of the room using the microphone
  • Generates a unique hash fingerprint of the sound profile
  • Future logins from a different acoustic environment (unfamiliar room) raise a trust warning
  • Implemented as a passive, transparent secondary factor — no user action required
  • Detects if someone else is using your credentials from a different location
🧬 Biometric Authentication
  • Uses Flutter's local_auth package (v2.1.0)
  • Android: Fingerprint via Android Keystore (hardware-backed)
  • iOS: Face ID / Touch ID via iOS Keychain
  • Used at: initial login, session unlock (auto-lock screen), key decryption
  • Biometric check is hardware-enforced — not bypassable by the app itself
🖼️ Steganography
  • Hides secret data inside ordinary image files
  • Uses LSB (Least Significant Bit) encoding in image pixels
  • An image containing a hidden message looks completely normal to the eye
  • Provides a covert channel for plausible deniability
  • Implemented in SteganographyService + SteganographyWidgets
💥 Self-Destructing Messages
  • Each message can have an expiresAt timestamp
  • When the expiry time passes, the server automatically sets status = EXPIRED and wipes encryptedContent and encryptedContentKey to null
  • A database index on idx_message_expires makes expiry queries efficient
  • The UI shows a countdown timer on expiring messages
📍 Location Verification
  • LocationVerificationService can restrict access to certain chats by geographic location
  • Useful for classified chats that should only be accessible from approved premises
  • Location data is verified server-side and never stored permanently
📸 Screenshot Protection
  • ScreenshotProtectionService sets the Flutter window's FLAG_SECURE flag on Android
  • Prevents the OS from including the app in the recent apps thumbnail
  • Blocks screenshot capture while in sensitive screens (chat, security settings)
